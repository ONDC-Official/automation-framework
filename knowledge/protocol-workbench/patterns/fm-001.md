# D-001 — on_X callback NACK (usually a generation symptom)

## Signature
- NACK on on_search/on_init/on_confirm: "message_id mismatch between X and on_X" (api-service ondcWorkbenchValidateContext, contextvalidator.go:~142). Only on_search/on_init/on_confirm are checked (others → null, mismatch masked).

## Quick checks (ordered)
1. **Is the mock-sent payload an ERROR payload?** Look at the action's payload: `error` instead of `message`, a FRESH context id, `error.code` = REQUIREMENTS_NOT_MET / REQUIREMENTS_CHECK_ERROR / generation error. → if yes, generation FAILED; the message_id NACK is a downstream symptom. Go to 2. (If no, go to 4.)
2. **Why did generation fail?** mock GENERATE_PAYLOAD_JOB log: "Requirements not met for action" (requirements.js returned valid:false; the `description` names the missing field) or "Mock payload generation failed / Syntax Error at line N" (base64 generator threw). REQUIREMENTS_CHECK_ERROR = requirements fn itself threw (often syntax).
3. **What does requirements/generate need vs what saveData provides?** Decode `mock.requirements`/`mock.generate` (base64) from config-service `/mock/playground`; list the `sessionData.*` it reads; grep the flow's `saveData` for those keys. A required field with NO saveData source = the bug (Gold_Loan_Offline: on_confirm needs `customer_*`, no `init` step saves them).
4. **Genuine echo mismatch:** message_id echo uses `sessionData.latestMessage_id` (MOST-recent), and `transaction_history` is unpopulated (Priority-2 lookup dead) → wrong whenever the predecessor isn't the immediately-last action (unsolicited on_status interleave, or w2w cross-side MOCK_DATA partition).

## Cause
- requirements() unmet ⇒ `buildErrorPayload` (fresh `createGenericContext` id, `error` not `message`) ⇒ generator never runs ⇒ message_id can't match. The protocol NACK is the surface; the root is generation.

## Fix locus
- primary (spec): add the missing `saveData` (e.g. confirm step saves `customer_name=$.message.order.fulfillments[0].customer.person.name` +phone/email) OR relax the requirements fn. Deploy via build-tools push-to-db + clear-flows (no api-service rebuild).
- secondary (mock): generate-response.ts — pin the predecessor's exact message_id from full per-action history (don't rely on latestMessage_id).

## Verify
- decode mock.requirements vs flow saveData; fresh flow → `on_confirm.message_id == confirm.message_id` (no error payload).
