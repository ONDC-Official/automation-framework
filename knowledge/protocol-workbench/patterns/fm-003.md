# D-003 — DYNAMIC_FORM (eKYC) completion stall

## Signature
- eKYC DYNAMIC_FORM submits; step sits INPUT-REQUIRED/PROCESSING, never reaches on_status; UI either "Form complete but failed to proceed" or a modal polling "Waiting for Form Submission" ~600s.

## Quick checks (ordered)
1. **Console error on submit?** `crypto.randomUUID is not a function` → secure-context issue → [[fm-006]] (the submit threw before /proceed).
2. **Did the BPP/UI session see completion?** `redis-cli -n 0 GET "form_completed:<UI sessionId>"`. Missing ⇒ the mock success page never hit /callback.
3. **Double-proceed?** mock log: real submission via `/forms/submit` then a second `/flows/proceed` with a different (placeholder) submission_id ⇒ the real submission gets overwritten.

## Cause
- Three independent, stacked causes (all fixed): (a) frontend sent a temporary `crypto.randomUUID()` submission_id and ALSO re-proceeded → overwrote the real submission, stranding the step (DOUBLE PROCEED); (b) mock `getSuccessHtml` just `window.close()`d — never navigated to api-service `/callback`, so `form_completed:{UI session}` (written only by the callbackredirect middleware via `redirection_url:{path}`) was never set → modal hangs; (c) in w2w the form runs on the BAP/counterparty session while the UI drives the BPP session → completion must bridge BAP-submit → BPP `form_completed`.

## Fix locus (all applied)
- mock idempotency guard (process-flow.ts dispatchTarget): if a real submission exists and the incoming submission_id differs → no-op (adr-0051).
- mock getSuccessHtml → redirect tab to `${formSession.subscriberUrl}/callback` (adr-0052) so callbackredirect writes `form_completed:{UI session}` + 302.
- frontend `crypto.randomUUID` → `uuidv4()` (adr-0048/d-006).

## Verify
- fresh flow: submit eKYC → success page redirects to /callback → `form_completed:{UI session}` written → modal closes → step advances to on_status.
