# D-004 — decode an api-service response status

## Signature
- You got a status off the api-service (ONIX) and need to know if it's expected. HTTP is **decoupled** from ACK/NACK (the ACK/NACK is in the body).

## Decode table
- **200 + `{ack:ACK}`** → success.
- **200 + `{ack:NACK, error}`** → schema/L1 validation failure with a well-formed context. `error.message` = `#### CODE … $.json.path`. Normal validation NACK. → fix the payload field at that JSONPath.
- **400** → bad request / missing forward target: e.g. `/mock/...` with no `bpp_uri` in context and no `?subscriber_url=` query param. (receiver guard, not router.)
- **412** → state-machine guard: "No active expectation found for transaction ID … as a seller_np" — out-of-sequence / unknown txn on a receiver path.
- **401** → signature (validateSign) failure on a txn path.
- **404** → not found (e.g. stale route — see [[fm-005]]).
- **500** → panic. Most common: malformed context (missing transaction_id ⇒ nil message_id) ⇒ `"Internal server error, MessageID: %!s(<nil>)"`. NOT a normal NACK.

## Cause / mechanism
- ONIX std-handler runs the module's step chain (short-circuit on error); the response builder maps error TYPE → HTTP code; ACK/NACK is always in the body. See [[onix-server]].

## Fix locus
- depends on the line above (payload field / context / sequence / signature / route).

## Verify
- re-send with the offending field/context fixed; expect 200 `{ack:ACK}`.
