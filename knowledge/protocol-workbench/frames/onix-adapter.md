# ONIX adapter (module + routing map)

`adapter.yaml` is the heart of the generated [[api-service]]: it declares modules (route blocks), each binding a URL path to an ordered plugin/step chain. GENERATED by [[api-service-generator]] (createAdapterFiles) from build.yaml, then rendered at container start by docker-entrypoint.sh (env substitution). Paths below illustrated for ONDC:FIS12/2.0.3 — structure is generated per spec.

## Modules (route blocks)
- **formReceiver** — `/.../form/html-form` (BAP) — router→addRoute → form_router → recorder /html-form.
- **standaloneValidator** — `/.../test/` (BAP) — schemaValidator → ondcValidator. `stateFullValidations:false`, NO signing. (UI Schema Validation Tool; L0/L1-syntax only.)
- **BapTxnReceiver** — `/.../seller/` (BAP receiver) — full chain (below). `stateFullValidations:true` + validateSign.
- **BppTxnReceiver** — `/.../buyer/` (BPP receiver) — symmetric to BapTxnReceiver.
- **callbackReceiver** — `/.../buyer/callback` (BPP) — form_router → callbackredirect middleware (HTML form completion → 302 redirect).
- **mockTxnCaller** — `/.../mock/` (role BAP, type caller) — proxies to mock; OUTBOUND signing (signer). Chain: cache→keyManager→networkobservability→router(mock_router)→schemaValidator→ondcValidator→ondcWorkbench→signer.

## Full receiver chain (Bap/BppTxnReceiver), ordered steps
ondcWorkbenchReceiver → addRoute → validateSchema → validateOndcPayload → ondcWorkbenchValidateContext → validateSign → validateOndcCallSave
(plugin order: cache → keyManager → networkobservability → router(np_router) → schemaValidator → ondcValidator → ondcWorkbench → signValidator)

## Routers
- np_router.yaml — Bap/BppTxnReceiver: route target from `$.cookies.mock_url`; proxies all actions when cookie present.
- mock_router.yaml — mockTxnCaller: target from `$.cookies.subscriber_url`; actAsProxy:true.
- form_router.yaml — form/callback: static `http://recorder-service:8090/html-form`; excludeAction:true.

## Action state machine
- transaction_properties.yaml defines allowed actions/state transitions (e.g. search allowed from null state); enforced by ondcWorkbench + ondc-validator stateful mode.

## Server execution (adr-0046)
- All modules run on the single `std` handler in [[onix-server]] — they differ by `steps`/`plugins`/`role`, not handler class. Steps run linearly, short-circuit on first error.
- Async forward: ActAsProxy=false ⇒ next hop fires as a post-response hook AFTER the client gets its ACK; workbench can inject a `custom-response-body` cookie (base64 ACK).

## Resolved (adr-0025; revised by adr-0038 runtime)
- Router has NO *default* fallback for jsonPath routes (router.go ~349-355). BUT a cookieless external call never reaches the router: `ondcWorkbenchReceiver` runs FIRST and resolves the forward target from **context** (`bap_uri`/`bpp_uri` by role; caller role falls back to `?subscriber_url=` query param — payloadutils.go:73-113). Missing target ⇒ **400/NACK from the receiver**, NOT a router 500.
- Live system-level statuses (FIS12 2.0.3, direct curl, no session): `/mock/search` (caller, no bpp_uri/subscriber_url) ⇒ **HTTP 400** "bpp_uri is missing in context and subscriber_url query param is also missing"; `/seller/on_search` (bap_uri present, no live txn) ⇒ **HTTP 412** "No active expectation found for transaction ID ... as a seller_np" (state-machine guard).
- Cookies (`subscriber_url`, `mock_url`, `header_validation`, `protocol_validation`, `flow_id`, `session_id`, `usecase_id`, `ttl_seconds`, difficulty knobs…) are set **OUTBOUND** by the workbench when it forwards (setRequestCookies, utils.go:39+) — i.e. the router's `$.cookies.*` is populated by an upstream workbench hop, not by the original external caller. See [[session-difficulty]].
